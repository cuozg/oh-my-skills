# Detail Task Plan

**Input**: Task/problem description, optional file paths, constraints, context.

**IMPORTANT**: Planning only — do NOT implement. Generate complete per-task code patches. Do NOT create tasks in Task System — user decides when to register tasks.

## Architecture: Templates vs Generated Files

**Templates** (`assets/templates/`) = internal boilerplate. **NEVER** access directly.
**Generated files** (`documents/plans/{plan-name}/`) = user-facing output.

| Template (internal) | Generated Output (user-facing) |
|---------------------|-------------------------------|
| `PLAN_OVERVIEW.html` | `documents/plans/{plan-name}/overview.html` |
| `PLAN_TASKS.html` | `documents/plans/{plan-name}/tasks.html` |
| `PLAN_PATCH.html` | `documents/plans/{plan-name}/patch.html` |
| `PLAN_PATCH_TEMPLATE.patch` | `documents/plans/{plan-name}/patches/TASK-{epic.task}.patch` |

### Output Folder Structure

```
documents/plans/{plan-name}/
├── overview.html          # Tab 1: Summary, architecture, risks, acceptance
├── tasks.html             # Tab 2: Epics, task breakdown, dependency graph
├── patch.html             # Tab 3: Per-task diff viewer (generated by script)
├── patches/               # Per-task .patch files
│   ├── TASK-1.1.patch
│   └── ...
└── tasks.json             # Task metadata for patch.html generation
```

## Generation Rules

1. Read template from `assets/templates/` for structure
2. Replace `[PLACEHOLDER]` (Overview/Tasks) or `{{PLACEHOLDER}}` (Patch) with plan content
3. Keep sticky nav as first `<body>` element; mark correct tab with `nav-tab-active`
4. Remove all `<!-- INSTRUCTION: ... -->` comments
5. Write to `documents/plans/{plan-name}/`
6. **No `<script>` tags or JavaScript** — pure HTML navigation only
7. Use sequential task labels (TASK-1.1, TASK-1.2) — not UUIDs
