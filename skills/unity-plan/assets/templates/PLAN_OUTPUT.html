🏗️ Architecture Overview
OLD ARCHITECTURE (Single Event)
Single static Dictionary<string, DailyBossResponseData> CachedDailyBossResponseData
└── All events share same cache
└── Data overwrites when switching events
└── No isolation between concurrent events
NEW ARCHITECTURE (Multi-Event)
DailyBossManager._eventCaches: Dictionary<string, EventStateCache>
├── Event 1 → EventStateCache (ID: "event_abc123")
│   ├── DifficultyResponses
│   ├── TierResponseData
│   ├── HighestDamageByTier
│   └── [7+ other state properties]
├── Event 2 → EventStateCache (ID: "event_xyz789")
│   └── [independent copy of all properties]
└── Event N → EventStateCache (ID: "event_nnn")
    └── [independent copy of all properties]

Key Benefits

  • Events maintain independent state
  • No data collisions when switching events
  • Server responses routed to correct event cache via eventId
  • Simple O(1) + O(1) nested dictionary lookups
  • Linear memory scaling (~10KB per event)
🎯 Technical Approach
  1. Move remaining single-event state into EventStateCache (or a new ActiveEventContext struct)
  2. Introduce an _activeEventIds list to track all currently active events
  3. Keep EventID as _focusedEventId — the event the player is currently interacting with (UI context)
  4. Add event-scoped overloads for SelectDailyBossDifficulty(), SetCurrentBossHP(), etc.
  5. Mark old single-event methods [Obsolete] and delegate to the new event-scoped versions
  6. Update callers progressively (foundation → non-breaking → breaking)
