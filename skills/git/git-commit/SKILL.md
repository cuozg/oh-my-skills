# Git Commit

Generate concise, meaningful commit messages based on file changes or user intent, then commit to the current branch.

## Purpose

Provide a dedicated, focused skill for generating clean git commit messages and executing commits with zero AI-related metadata.

## Input

- **Required**: Either file changes (detected via `git diff`) or explicit user request describing changes
- **Optional**: Commit scope/context (e.g., "WP_Unity_80 project", "WWE Champions client")
- **Optional**: Branch name to verify before committing (safety check)

## Commit Message Format

```
<type>: <subject>

<body (optional)>
```

### Types

| Type | Usage |
|------|-------|
| `feat` | New feature |
| `fix` | Bug fix |
| `refactor` | Code restructuring without feature changes |
| `docs` | Documentation updates |
| `chore` | Build, tooling, dependency updates |
| `test` | Test additions/updates |
| `perf` | Performance improvements |
| `style` | Code style (formatting, no logic change) |

### Subject Line

- **Length**: 50 characters or less
- **Style**: Imperative mood ("add", "update", "fix", not "added", "updating", "fixed")
- **Capitalization**: Lowercase
- **Period**: No period at end
- **Content**: Clear, atomic change description

### Body (Optional)

- Explain **what** and **why**, not **how**
- Use bullet points for multiple changes
- Wrap at 72 characters
- Separate from subject with blank line

## Restrictions

- **NEVER** add co-author, committer, or AI tool metadata
- **NEVER** add `Co-authored-by:`, `Tool-generated-by:`, or similar lines
- **NEVER** push to remote — only commit to current branch
- **NEVER** include implementation details or code snippets in message
- **DO** use imperative mood
- **DO** keep messages concise and meaningful
- **DO** use bullet points for multiple changes
- **DO** verify branch before committing

## Workflow

### 1. Analyze Changes

```bash
git diff --stat           # Show changed files
git diff --cached --stat  # Show staged changes
git status                # Overall state
```

### 2. Generate Message

Based on changes:
1. Identify **type** (feat, fix, refactor, etc.)
2. Write **subject** (clear, atomic, <50 chars)
3. Write **body** if changes are complex (explain why, use bullets)

### 3. Commit

```bash
git add <files>                           # Stage specific files
git commit -m "type: subject"             # Simple commit
git commit -m "type: subject" -m "body"   # With body
```

### 4. Verify

```bash
git log --oneline -1    # Confirm commit was created
```

## Examples

### Simple fix

```
fix: resolve null reference in PlayerManager initialization
```

### Feature with multiple files

```
feat: add daily boss event handler system

- Implement BaseHUDEventHandler subclass for Daily Boss events
- Extend HUDEventBus to route new event types
- Add unit tests for handler initialization and event dispatch
```

### Refactoring

```
refactor: extract data adapters from UI controllers

- Create IDataAdapter interface and concrete adapters
- Update UIManager to use adapters instead of direct data access
- Improve testability through dependency injection
```

### Documentation

```
docs: add architecture documentation for NewHomeHUD system
```

## Anti-Patterns

| Bad | Good |
|-----|------|
| "Fixed stuff" | `fix: resolve null reference in PlayerManager` |
| "Updated code" | `refactor: simplify event handling logic` |
| "WIP" or "test" | `feat: add daily boss notification system` |
| `Co-authored-by: Claude AI` | (no metadata) |
| `Tool-generated-by: oh-my-opencode` | (no metadata) |
| Multiple unrelated changes in one commit | One type per commit, atomic changes |
